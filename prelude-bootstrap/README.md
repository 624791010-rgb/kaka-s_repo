# Prelude Bootstrap

> A reusable skill for bootstrapping AI-facing project prelude files.  
> 用来给工程创建一套稳定的 `ai/` 前置文件体系，方便多窗口、多 agent、长周期迭代协作。

---

## 1. 这是什么

`Prelude Bootstrap` 是一个面向 AI 工程开发的 skill。

它解决的问题不是“写业务代码”，而是更前面的那一步：

- 新项目一上来，AI 不知道先读什么
- 项目做了几轮后，知识散落在聊天、文档、代码和人脑里
- 开新窗口做 PIV 时，AI 很难快速接上上下文
- 交接包名字会变，规则会变，但入口不能乱

这个 skill 的目标是：

1. 先浏览工程现状
2. 识别真实技术栈、主要模块、关键流程、共享高风险文件
3. 在项目中创建或规范化一套稳定的 `ai/` 层
4. 让后续 agent 进入项目时，知道：
   - 先读什么
   - 哪些是长期规则
   - 哪些是本轮交接
   - 哪些文件高风险不能乱碰

---

## 2. 它适合什么场景

这个 skill 适合：

- 新项目刚开始，需要建立 AI 协作基础设施
- 老项目已经做了一段时间，但没有统一前置文件
- 多窗口并行开发，需要标准化 handoff
- 外包项目接手后，需要先把知识层沉淀下来
- 准备做 PIV 切分、隔离开发、分阶段提交

它不适合：

- 只想写一个页面
- 只想修一个很小的 bug
- 项目已经有成熟稳定的 `ai/` 知识层，而且不想改

---

## 3. 它会生成什么

默认会围绕 `project-root/ai/` 建立一套前置文件结构。

典型结构：

```text
project-root/
  ai/
    prd.md
    global-rules.md
    reference/
      git-workflow.md
      project-map.md
      qa-checklist.md
      style-guide.md
      component-guide.md
      api-conventions.md
      piv-roadmap.md
      handoff-workflow.md
      handoff-template.md
    commands/
      prime.md
      plan-feature.md
      commit.md
    archive/
      latest.md
      handoff_0.md
```

核心思想：

- `prd.md`：产品目标和范围入口
- `global-rules.md`：长期规则
- `reference/*`：专项参考
- `commands/*`：固定动作入口
- `archive/*`：每一轮的交接包

## 4. AI 工程开发推荐流程

这个 skill 背后的方法论不是“先写代码”，而是：

```text
脑暴 → 调研 → 追问 → PRD → Rules → Prime → PIV → Commit → Evolve
```

下面是每一步的作用。

### 4.1 脑暴

目标：把想法从模糊需求变成可以讨论的问题。

要做的事：

- 明确用户是谁
- 明确要解决什么问题
- 明确为什么值得做
- 明确最小版本先做什么，不做什么

产出不是代码，而是候选方向。

### 4.2 调研

目标：避免拍脑袋。

要做的事：

- 看竞品
- 看已有系统
- 看工程现状
- 看现有数据和资源约束

这一步常见错误是：

- 还没看项目就开始设计
- 还没看代码就开始重构

### 4.3 追问

目标：把高影响歧义问掉。

典型追问包括：

- 这个功能给谁用
- 首发闭环是什么
- 后台要不要同步做
- 假数据能不能接受
- 需要并行开发吗

追问不是为了拖延，而是为了减少返工。

### 4.4 PRD

目标：把原始需求固定成业务范围。

PRD 至少要回答：

- 做什么
- 不做什么
- 核心流程是什么
- 验收标准是什么
- 哪些是当前版本，哪些以后再说

PRD 不应该混入太多工程细节。

### 4.5 Rules

目标：把长期约束写下来。

Rules 负责的是：

- 技术栈
- 工作方式
- 提交流程
- 高风险文件
- 公共契约优先级
- 启动阅读顺序

Rules 不负责写详细产品流程，也不负责写本轮 handoff。

### 4.6 Prime

目标：让新窗口开工前先进入同一个上下文。

Prime 通常规定：

1. 先看 git 状态
2. 先读 global rules
3. 先读 PRD 入口
4. 先读最新 handoff
5. 再读专项参考文件

Prime 的本质是“固定启动顺序”。

### 4.7 PIV

目标：把大工程拆成可独立验收的纵向闭环。

PIV 不是目录切分，也不是按前后端横着切。

一个好的 PIV 应该：

- 有明确目标闭环
- 有前置依赖
- 有清楚的验收标准
- 有尽量小的冲突面
- 可以单独提交、单独 review、单独回滚

### 4.8 Commit

目标：让每轮变化可追踪、可回退、可交接。

原则：

- 一轮一个意图
- 文档和代码尽量分开提交
- 不把无关脏改动顺手带走
- 能 `git add -p` 就不要整文件盲提

### 4.9 Evolve

目标：工程持续进化，而不是一次性做完。

Evolve 包括：

- 补 handoff
- 更新 latest
- 调整 rules
- 新增专项 reference
- 继续拆 PIV

不是每轮都要重写体系，但每轮都应该让体系更稳一点。

## 5. 怎么从脑暴产出 PRD

推荐做法：

### Step 1. 先写原始目标

用最简单的话写：

- 用户是谁
- 他要做什么
- 他为什么要这样做

### Step 2. 再写最小闭环

问自己：

- 如果只做一版最小可用版本，用户从进入到完成，最短路径是什么

### Step 3. 写范围边界

至少分三类：

- 本轮一定做
- 本轮明确不做
- 后续可能做

### Step 4. 写验收标准

不要写“体验更好”这种空话，要写：

- 用户点击什么
- 系统返回什么
- 数据写到哪里
- 后台是否能看到

## 6. 怎么制定各种 Rules

推荐把 rules 分成三层：

### 6.1 Global Rules

长期、常驻、跨任务规则。

例如：

- 技术栈
- Git 规则
- 高风险文件
- 文档入口

### 6.2 Reference Rules

专项规则。

例如：

- API 约定
- UI 风格
- QA 清单
- Handoff 流程

### 6.3 Iteration Rules

只属于当前轮次，不该进 global rules。

例如：

- 本轮 PIV 范围
- 本轮冻结项
- 本轮交接包

常见错误：

- 把所有东西都塞进 global-rules
- 把本轮临时问题写成长期规则
- 不给新窗口固定入口

## 7. 怎么做 PIV 隔离

这是这套方法里最重要的一部分。

### 7.1 PIV 的定义

PIV = 一个可以独立闭环、独立验收、独立提交的纵向子板块。

它应该包含：

- 页面
- 接口
- 数据
- 状态
- 验收

而不是只改一个目录。

### 7.2 切 PIV 的原则

优先按“用户闭环”切，不按目录切。

例如：

- 登录链路
- 学生认证
- 资料与偏好
- 问卷与画像
- 恋爱匹配主链
- 聊天
- 任务与积分
- 邀请裂变

### 7.3 一个 PIV 应该写清什么

至少写清：

- 目标闭环
- 前置依赖
- 涉及文件
- 不该碰的文件
- 验收标准
- 风险点
- 是否可并行

### 7.4 为什么要隔离

因为 AI 和多人协作最大的风险就是：

- 大家都在改共享文件
- 改完讲不清边界
- 交接时不知道这轮到底做了什么

PIV 隔离就是把这些风险降下来。

### 7.5 PIV 与 handoff 的关系

每做完一轮 PIV，最好都产出：

- 一次代码快照提交
- 一份 handoff 文档
- `archive/latest.md` 更新

这样下个窗口接手时，才不会重新考古。

## 8. 如何使用这个 skill

当用户说这些话时，适合触发：

- 帮我给这个项目补一套前置文件
- 帮我做 AI 协作知识层
- 帮我建 `ai/` 目录
- 帮我整理 handoff 体系
- 帮我标准化 PRD / rules / prime / PIV 文档

触发后，agent 应该：

1. 先浏览工程
2. 识别真实栈和关键文件
3. 识别已有 docs
4. 创建或规范化 `ai/` 层
5. 报告新建了哪些文件、用了哪些现有文档做真相源

---

## 9. 设计原则

这个 skill 的核心原则只有 5 个：

1. **先看工程，再写文档**
2. **文档入口稳定，交接文件名可以变化**
3. **长期规则和本轮交接分开**
4. **前置文件服务于多窗口衔接**
5. **PIV 是纵向闭环，不是目录切分**
